> **Scope note:** RFC 3253 defines two distinct things:
>
> 1. **The `REPORT` method** (§3.6) — a generic WebDAV method used by CalDAV, CardDAV, and ACL. It is defined here but applies to any WebDAV server, independent of versioning support. These requirements are cross-cutting and testable.
>
> 2. **The DeltaV versioning feature set** — VERSION-CONTROL, CHECKOUT, CHECKIN, version history, workspaces, etc. These are optional extensions requiring explicit server opt-in via the `version-control` compliance token in OPTIONS. Essentially no modern CalDAV/CardDAV server implements DeltaV versioning. All versioning requirements are deferred.

## §3.6 — REPORT Method (Cross-Cutting)

The REPORT method is defined in RFC 3253 but is a generic WebDAV extension used by RFC 4791 (CalDAV), RFC 6352 (CardDAV), RFC 3744 (ACL), and RFC 6578 (sync). These requirements apply to **any server supporting REPORT**, regardless of versioning support.

| Req | Requirement | Current Test | Status |
|-----|-------------|--------------|--------|
| R-01 | REPORT MUST NOT modify the content or dead properties of any resource | — | not covered |
| R-02 | Server MUST return 207 Multi-Status when a `Depth` header is included in a REPORT request | — | not covered |
| R-03 | Server MUST fail with `DAV:supported-report` precondition (403) when an unsupported report type is requested | — | not covered |
| R-04 | `DAV:supported-report-set` property MUST be present and list all supported reports | — | not covered |

**Missing tests:**
- `rfc3253.report-unsupported-returns-403` — REPORT request using an unknown/unsupported report element returns 403 with `DAV:supported-report` precondition in the body
- `rfc3253.report-does-not-modify-state` — issuing a calendar-query REPORT twice returns the same result (REPORT is idempotent and non-modifying)
- `rfc3253.supported-report-set-present` — PROPFIND on a calendar collection returns `DAV:supported-report-set` *(note: also tested in rfc4791 and rfc6352 suites; may be duplicate)*

> **Note:** `DAV:supported-report-set` is also validated in `rfc4791.supported-report-set` and `rfc6352.supported-report-set`. The `rfc3253` suite should focus on the generic REPORT method contract; CalDAV/CardDAV-specific report advertisement is better placed in those suites.

## §4 — VERSION-CONTROL Feature

| Req | Requirement | Current Test | Status |
|-----|-------------|--------------|--------|
| R-05 | Server MUST include `version-control` in `DAV` OPTIONS header if versioning is supported | — | deferred |
| R-06 – R-50 | All VERSION-CONTROL, CHECKOUT, CHECKIN, version history, workspace, auto-version, and fork requirements | — | deferred |

**Deferred reason (R-05 – R-50):** tagged `versioning` — DeltaV versioning is an optional extension not implemented by essentially any modern CalDAV/CardDAV server. Excluded from all protocol bundle runs. Would require a dedicated `--tag versioning` flag and server discovery of `version-control` in the DAV header before running.
