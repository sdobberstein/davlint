# RFC 6578 — WebDAV Sync-Collection

**Suite:** `rfc6578` | **Test file:** `internal/suites/rfc6578/rfc6578.go`

Defines the `DAV:sync-collection` REPORT, sync tokens as URIs, initial and subsequent sync semantics (added/changed/removed members), result truncation, and the `DAV:sync-token` live property.

## Deferred

| Requirement | Reason |
|-------------|--------|
| Client must treat sync-token as opaque (R-02) | Client-side rule |
| Malformed request body → 4xx (R-05/R-06) | Server error responses vary too widely to assert deterministically |
| sync-level:infinite + unsyncable child → 403 (R-15/R-16, R-23–R-25b) | Requires nested addressbook collections |
| Server MUST limit token invalidation window (R-18) | Not triggerable deterministically |
| Client must include DAV:sync-level (R-21) | Client-side; all requests already include it |
| Cross-level token reuse (R-22) | Requires sync-level:infinite support |
| Collection-type members in sync (R-28, R-30, R-33, R-36) | Requires nested collection members |
| Client assumes cascade on collection removal (R-37) | Client-side rule |
| Client must handle 507 (R-40) | Client-side; server behavior covered by R-38 |
| ACL-triggered change/removal (M01/M02) | Requires ACL manipulation; non-deterministic |
| Server MAY truncate without client request (M03) | Server-internal policy |

## Gaps

| Area | What's missing |
|------|----------------|
| Invalid token error body | `invalid-token-rejected` only checks status; body must contain `DAV:error` with `DAV:valid-sync-token` |
| sync-token protected | PROPPATCH setting `DAV:sync-token` must return 403 |
| sync-token consistency | PROPFIND value must equal the token from an immediately preceding REPORT |
| Updated resource in subsequent sync | PUT updating an existing resource must appear as changed (R-31) |
| Truncation follow-up | Token from a truncated (507) response must be usable in a follow-up sync |
| Truncation error body | 507 response should include `DAV:error` with `DAV:number-of-matches-within-limits` |
| If-header conformance across methods | Current tests cover PROPFIND only; PUT/DELETE/MKCOL with valid/stale sync-token not tested |

## Depth notes

- `invalid-token-rejected`: 4xx status checked; error body not inspected
- `subsequent-sync-changed`: only covers newly added resources; updated resources (ETag change) not tested
- `client-limit`: 507 sub-status presence checked; `DAV:error` body and follow-up sync behavior not verified
- `if-header-valid-token` / `if-header-stale-token`: PROPFIND only; RFC 6578 §5 applies to any conditional request method
