# RFC 6352 — CardDAV

**Suite:** `rfc6352` | **Test file:** `internal/suites/rfc6352/rfc6352.go`

Defines CardDAV: address book collections, address object resources (vCard PUT/GET with preconditions), the addressbook-query and addressbook-multiget REPORTs, collations, and unauthenticated access enforcement. 63 tests covering all testable requirements.

## Deferred

| Requirement | Reason |
|-------------|--------|
| WebDAV ACL support (R-03) | Tagged `acl` — excluded by default |
| TLS support (R-04) | TLS-level; out of scope for HTTP black-box testing |
| Server MAY restrict addressbook creation locations (R-33) | Internal server policy; not observable |
| Server MAY reject changes to its own X- properties (R-47) | Permissive; server-specific |
| MUST NOT return strong ETag when data normalized (R-52) | Only observable if server normalizes on PUT; hard to trigger deterministically |
| Server MAY advertise reports on ordinary collections (R-62) | Permissive |
| Server MAY truncate results (R-90) | Permissive |
| CR omission / partial vCard validity in responses (R-105/106) | Formatting/rendering detail |
| Result truncation structure: R-91–R-93 (507 sub-status, DAV:error body, partial results) | Not triggerable deterministically from client side |

## Gaps

None — all testable requirements are covered.

## Depth notes

- All PUT preconditions (supported-address-data, valid-address-data, no-uid-conflict, max-resource-size): one test each; combinations not tested
- COPY/MOVE preconditions: addressbook-collection-location-ok and no-uid-conflict each covered for both verbs
- addressbook-query and addressbook-multiget: response structure spot-checked; not exhaustively verified across filter variants
- Collation tests exercise the query round-trip but do not verify output ordering
- expand-property REPORT is conditional on server advertisement in supported-report-set (SHOULD, not MUST)
- addressbook-multiget Depth:1 → if server accepts it (permissive), test passes trivially
