# RFC 6764 — Service Discovery for CalDAV and CardDAV

**Suite:** `rfc6764` | **Test file:** `internal/suites/rfc6764/rfc6764.go`

Defines how clients locate CalDAV and CardDAV services: well-known URIs (`/.well-known/carddav`, `/.well-known/caldav`), the PROPFIND bootstrapping chain (context path → current-user-principal → home set), and authentication requirements.

## Deferred

| Requirement | Reason |
|-------------|--------|
| TLS certificate verification: SRV-ID, DNS-ID (R-07-5/R-07-6, R-08-7/R-08-8) | Requires TLS-level certificate inspection |
| Server MUST NOT accept SSL 2.0 (R-08-9) | Requires TLS-level protocol negotiation |
| All §3–§6 DNS/client-side requirements | SRV lookup, TXT records, bootstrapping algorithm — DNS-level or client-side rules not observable via HTTP |

## Gaps

| Area | What's missing |
|------|----------------|
| Email-format authentication (R-07-1, SHOULD) | Authenticate with email-format username; verify principal URL returned — requires specific credential config |
| CalDAV equivalents | `/.well-known/caldav` redirect, `CALDAV:calendar-home-set` PROPFIND — blocked on rfc4791 suite |

## Depth notes

- Well-known tests verify the redirect status and Location header but do not follow the redirect chain to confirm it resolves
- current-user-principal and addressbook-home-set checked for presence; property values are only loosely validated (href must be non-empty)
- `rfc6764.dav-header` (OPTIONS returns `DAV: 1, 2, access-control`) is placed here as a discovery chain verification; the normative source is RFC 4918 §10.1
