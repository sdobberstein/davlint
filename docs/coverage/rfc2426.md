# RFC 2426 — vCard 3.0

**Suite:** `rfc2426` | **Test file:** `internal/suites/rfc2426/rfc2426.go`

Defines the vCard 3.0 format. Tests here validate server behavior observable via HTTP: rejecting malformed vCards on PUT and verifying data preservation through GET round-trips. Pure client-side authoring rules are deferred. 18 tests covering all testable requirements.

## Deferred

| Requirement | Reason |
|-------------|--------|
| vcard-type inline value (`AGENT` property) line termination and escaping (R-08/R-09) | AGENT/embedded-vCard type is extremely rare in modern CardDAV servers; out of scope |
| PRODID SHOULD use ISO 9070 uniqueness (S-03) | Client-side authoring rule; not server-enforceable |
| vCard SHOULD be transported securely (S-04) | TLS-level; outside HTTP black-box scope |
| URL SHOULD be used when data vitality important (S-05) | Application-level authoring guidance |

## Gaps

| Area | What's missing |
|------|----------------|
| Binary inline properties | PHOTO, LOGO, SOUND, KEY with inline data require `ENCODING=b`; not tested (R-06, R-07, R-16, R-20, R-21, R-24) |
| TZ format | `TZ:+0530` (compact) vs `TZ:+05:30` (ISO 8601 extended) (R-10) |
| EMAIL TYPE preservation | `TYPE=INTERNET` preserved on 3.0 round-trip (R-18); cross-reference to rfc6352.email-roundtrip |
| Server output folding | Lines >75 chars should be folded in GET response (S-02, SHOULD) |

## Depth notes

- BEGIN/END delimiter tests and line folding are conformance ✓ — full round-trip verified
- Semicolon escape tests verify acceptance and preservation only; the reject path (unescaped `;` in non-compound field) is not tested
- ADR structure roundtrip verifies separator presence but not full component count
- `EMAIL;TYPE=INTERNET` conformance is tested in the rfc6352 suite via content negotiation (4.0 → 3.0); not tested here for native 3.0 round-trips
