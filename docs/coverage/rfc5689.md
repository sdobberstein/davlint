# RFC 5689 — Extended MKCOL

**Suite:** `rfc5689` | **Test file:** `internal/suites/rfc5689/rfc5689.go`

Extends plain MKCOL (RFC 4918 §9.3) to accept a request body specifying the new collection's resource type and initial properties. The standard mechanism for creating CardDAV address books and CalDAV calendars without protocol-specific methods.

## Deferred

| Requirement | Reason |
|-------------|--------|
| `DAV:mkcol-response` body format (R-08) | MAY — server may return this; not required |
| MKCALENDAR equivalence (R-11) | Conditional on CalDAV suite (rfc4791) — deferred to CalDAV milestone |

## Gaps

| Area | What's missing |
|------|----------------|
| Calendar collection creation | `extended MKCOL + CALDAV:calendar` returns 201 — blocked on rfc4791 |
| MKCALENDAR equivalence | Calendar via extended MKCOL has same resourcetype as one via MKCALENDAR — blocked on rfc4791 |

## Depth notes

- All 10 covered requirements are `lint ✓` — 201 status checked but resulting collection properties not verified via PROPFIND in most cases
- `mkcol-property-failure-atomic` verifies the request fails (not 201) but does not confirm via GET/PROPFIND that no collection was created
