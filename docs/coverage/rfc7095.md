# RFC 7095 — jCard: The JSON Format for vCard

**Suite:** not yet created

Defines a JSON representation of vCard 4.0 using the media type `application/vcard+json`. A CardDAV server MAY support this alongside `text/vcard` via HTTP content negotiation (Accept / Content-Type headers).

## Standalone suite verdict

**Low priority — create only if target servers advertise support.** jCard support is a MAY in CardDAV; most production servers (Radicale, Baikal, Cyrus, Apple) do not implement it. A suite would:

1. GET a contact with `Accept: application/vcard+json` → if 406, skip (not supported)
2. If 200, validate response is valid JSON array with correct jCard structure (`["vcard", [...]]`)
3. PUT with `Content-Type: application/vcard+json` → server accepts or returns 415

The skip-if-unsupported pattern makes this low-value for the current target set. Revisit when a target server implementing jCard is identified.

## Deferred

| Requirement | Reason |
|-------------|--------|
| Full jCard schema validation | Would require a JSON schema validator; out of scope for black-box HTTP testing |
| Round-trip fidelity (text/vcard → jCard → text/vcard) | Implementation-internal format conversion |
