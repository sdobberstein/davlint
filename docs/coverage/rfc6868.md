# RFC 6868 — Parameter Value Encoding in iCalendar and vCard

**Suite:** not yet created

Defines caret (`^`) encoding for special characters in iCalendar and vCard property *parameter* values — specifically `^n` (newline), `^^` (literal caret), and `^'` (double-quote). Without this, parameter values cannot safely contain newlines or double-quotes, which causes interoperability problems for e.g. `LABEL` or `CN` parameters.

## Standalone suite verdict

**Yes — worth a suite.** Short spec (one encoding rule), testable in a black-box manner via PUT→GET round-trips, applies to both vCard and iCalendar. A server that does not understand `^` encoding will either reject the PUT or silently corrupt the parameter value. Could live as a sub-section of `rfc2426`/`rfc6350` or as a standalone `rfc6868` suite; standalone is cleaner since the rule applies to both vCard versions.

## Testable requirements

| Area | Test sketch |
|------|-------------|
| `^^` round-trip (vCard) | PUT `FN;X-LABEL=path^^to^^file:Alice` → GET contains caret or decoded equivalent |
| `^n` round-trip (vCard) | PUT `FN;X-LABEL=line1^nline2:Alice` → GET contains newline in param or decoded equivalent |
| `^'` round-trip (vCard) | PUT `FN;X-LABEL=say^'hello^':Alice` → GET contains double-quote in param |
| Invalid caret sequence rejected or passed through | PUT `FN;X-LABEL=bad^x:Alice` → server accepts (unknown sequences pass through per §4) |

## Deferred

| Requirement | Reason |
|-------------|--------|
| iCalendar `^` encoding | Blocked on rfc5545/rfc4791 suite existing |
| Server MUST re-encode on output | Output encoding format is implementation-defined; a server may decode to native form and re-encode differently |
