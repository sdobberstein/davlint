# RFC 7232 — HTTP Conditional Requests

**Suite:** `rfc7232` | **Test file:** `internal/suites/rfc7232/rfc7232.go`

Defines HTTP conditional request headers (If-Match, If-None-Match, If-Modified-Since, If-Unmodified-Since) and validator semantics (ETags, Last-Modified). Critical for safe concurrent writes in CalDAV/CardDAV — RFC 6352 §5.3.4 and RFC 4791 §5.3.4 require servers to enforce these semantics. 25 tests covering all testable requirements.

## Deferred

| Requirement | Reason |
|-------------|--------|
| SHOULD update weak ETags when representation becomes unacceptable substitute (R-02) | Implementation-internal; effect not distinguishable in black-box testing |
| MUST NOT send Last-Modified later than response Date (R-04) | Requires sub-second clock comparison between response headers |
| MUST replace future modification dates with origination time (R-05) | Requires clock manipulation or future-timestamped resources |
| MAY send weak ETags when strong infeasible (R-07) | MAY — no server obligation; not testable |
| MUST evaluate preconditions after normal checks, before method action (R-30) | Implementation-internal evaluation ordering |

## Gaps

None — all testable requirements are covered.

## Depth notes

- Strong ETag check (no `W/` prefix) simultaneously covers RFC 6352 §5.3.4
- `precondition-ignored-for-error-response` (R-31) and `not-modified-has-validator` (R-28) are `Should` — R-31 because many servers return 412 instead of 404 on absent resources, R-28 because it only checks that at least one validator header is present
- `if-modified-since-non-get-ignored` (R-22) tests the failure mode where a server incorrectly applies IMS to PUT and returns non-2xx
- `evaluation-order-im-before-inm` (R-33) tests ordering by sending a request where evaluating If-None-Match first would succeed but If-Match (evaluated first per §6) fails
