# RFC 2426 — vCard MIME Directory Profile (vCard 3.0)

**Test file:** `internal/suites/rfc2426/rfc2426.go`
**Tests registered:** 11
**Requirements extracted:** ~29 (~18 directly HTTP-testable)

Coverage statuses:
- `conformance ✓` — all meaningful variations tested
- `lint only` — representative test exists; conformance variants missing
- `not covered` — no test exists
- `deferred` — intentionally out of scope with reason

> **Scope note:** RFC 2426 defines the vCard 3.0 format, not HTTP protocol. Tests in this suite validate server behaviour *observable via HTTP*: rejecting malformed vCards, accepting correctly-formatted content, and preserving data through round-trips. Pure client-side authoring rules (how a sender MUST format data it produces) are deferred unless a server is also expected to validate them on receipt.

## §2.3 — Text Value Encoding

| Req | Requirement | Current Test | Status |
|-----|-------------|--------------|--------|
| R-04 | SEMI-COLON in a text value MUST be backslash-escaped | `rfc2426.semicolon-escape-accepted` | lint only |
| R-05 | Predefined type values MUST NOT repeat in COMMA-separated lists (except N, NICKNAME, ADR, CATEGORIES) | — | not covered |

**Conformance gap (R-04):** `semicolon-escape-accepted` verifies only that a correctly escaped `;` in FN is accepted and preserved. It does not test other text-type properties (e.g., NOTE, ORG), nor does it verify that a vCard with an *un*escaped literal `;` in a non-compound field is rejected or sanitised.

**Missing tests:**
- `rfc2426.semicolon-escape-roundtrip-note` — backslash-escaped `;` in NOTE survives round-trip
- `rfc2426.reject-duplicate-type-value` — PUT a vCard with a duplicate predefined type value in a plain text property is rejected

## §3.3.2 — EMAIL

| Req | Requirement | Current Test | Status |
|-----|-------------|--------------|--------|
| R-18 | EMAIL TYPE parameter MUST include `INTERNET` in vCard 3.0 | `rfc2426.email-roundtrip` | lint only |

**Conformance gap:** `email-roundtrip` verifies that a vCard 4.0 stored and served as 3.0 includes `TYPE=INTERNET`. It does not test that a 3.0 vCard PUT with `TYPE=work` (no INTERNET) round-trips correctly, or that `INTERNET` is preserved when the source is already 3.0.

**Missing test:**
- `rfc2426.email-internet-type-preserved-v3` — PUT a vCard 3.0 with `EMAIL;TYPE=INTERNET,work:...`; GET as 3.0 returns INTERNET in TYPE list
