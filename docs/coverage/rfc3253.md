# RFC 3253 — DeltaV (Versioning Extensions to WebDAV)

**Suite:** `rfc3253` | **Test file:** `internal/suites/rfc3253/rfc3253.go`

Defines two things: (1) the **REPORT method** (§3.6) — a generic WebDAV method used by CalDAV, CardDAV, ACL, and sync; and (2) the **DeltaV versioning feature set** — practically never implemented by modern CalDAV/CardDAV servers. 2 tests covering all testable generic-REPORT requirements.

## Deferred

| Requirement | Reason |
|-------------|--------|
| All versioning requirements (§4+): VERSION-CONTROL, CHECKOUT, CHECKIN, version history, workspaces, auto-version, fork restrictions | Tagged `versioning` — DeltaV is not implemented by essentially any modern CalDAV/CardDAV server; excluded from all bundle runs |

## Gaps

None — all testable generic REPORT requirements are covered.

## Depth notes

- `unsupported-report-type`: sends a REPORT body with `<x:unknown-report xmlns:x="urn:davlint:test"/>` — a namespace no server will recognise; verifies 403 and `DAV:supported-report` in the error body. `DAV:supported-report-set` population is already tested in rfc6352 and rfc6578 suites.
- `report-idempotent`: two identical initial sync-collection REPORTs on an empty collection; the sync-token returned by both MUST be equal, proving no state mutation.
