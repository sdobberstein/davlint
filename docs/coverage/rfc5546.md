> **Scope note:** RFC 5546 is a transport-independent protocol spec — it defines the *semantics* of calendar scheduling messages (REQUEST, REPLY, CANCEL, ADD, REFRESH, COUNTER, DECLINECOUNTER) and how their iCalendar content must be structured. It does not define any HTTP methods, WebDAV properties, or request/response shapes.
>
> RFC 5546 is not independently testable via HTTP. Its rules manifest as server behaviour only through the CalDAV scheduling API defined in RFC 6638: when a CalDAV server processes incoming scheduling messages it MUST follow RFC 5546 rules (RFC 6638 §4). Tests for RFC 5546 conformance therefore belong in the **rfc6638 suite**, not here.
>
> **Relationship to other specs:**
> - RFC 5545 (iCalendar format) is to RFC 4791 (CalDAV) as RFC 5546 (iTIP semantics) is to RFC 6638 (CalDAV Scheduling).
> - No standalone rfc5546 suite is needed; this document exists for completeness.
