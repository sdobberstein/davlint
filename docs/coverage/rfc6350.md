# RFC 6350 — vCard Format Specification (vCard 4.0)

**Test file:** `internal/suites/rfc6350/rfc6350.go`
**Tests registered:** 6
**Requirements extracted:** ~35 (~20 directly HTTP-testable)

Coverage statuses:
- `conformance ✓` — all meaningful variations tested
- `lint only` — representative test exists; conformance variants missing
- `not covered` — no test exists
- `deferred` — intentionally out of scope with reason

> **Scope note:** RFC 6350 defines the vCard 4.0 format. Like RFC 2426, tests here validate server behaviour observable via HTTP: rejecting malformed vCards on PUT and verifying correct content on GET. Pure authoring rules (how a sender MUST format data it produces) are deferred unless a server is expected to validate them on receipt.
>
> **Suite placement note:** Three of the six registered tests (`get-accept-v4`, `get-accept-v3`, `supported-address-data`) exercise RFC 6352 §6.5.3 content-negotiation and `CARDDAV:supported-address-data` property behaviour. They are valid regression tests but do not map to RFC 6350 normative requirements. `photo-size-limit` enforces a server-specific size policy, also RFC 6352 territory (§6.2.5). These are called out in the table below.
