# RFC 5546 — iTIP (iCalendar Transport-Independent Interoperability Protocol)

**Suite:** not needed as a standalone suite

Defines the semantics of calendar scheduling messages (REQUEST, REPLY, CANCEL, ADD, REFRESH, COUNTER, DECLINECOUNTER) and how their iCalendar content must be structured. RFC 5546 has no HTTP methods, WebDAV properties, or request/response shapes of its own — it is a transport-independent semantic spec.

iTIP rules manifest as server behavior only through the CalDAV scheduling API defined in RFC 6638: when a CalDAV server processes incoming scheduling messages it MUST follow RFC 5546 rules (RFC 6638 §4). Observable effects — SCHEDULE-STATUS parameters, PARTSTAT updates, SEQUENCE values in stored resources — are tested in the **rfc6638 suite**.
