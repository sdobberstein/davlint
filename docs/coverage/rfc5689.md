> **Context:** RFC 5689 extends plain MKCOL (RFC 4918 §9.3) to accept a request body that specifies the new collection's resource type and initial properties. This is the standard way to create typed collections — CalDAV calendars and CardDAV address books — without requiring protocol-specific methods (MKCALENDAR). RFC 4791 §5.3.1 and RFC 6352 §6.3.1 both reference extended MKCOL as the preferred creation mechanism.

## §4 — Integration with CalDAV and CardDAV

RFC 5689 §4 states that when a server supports both extended MKCOL and a protocol-specific creation method (e.g. MKCALENDAR for CalDAV), it MUST support using extended MKCOL as an equivalent alternative.

| Req | Requirement | Current Test | Status |
|-----|-------------|--------------|--------|
| R-11 | If server supports MKCALENDAR (RFC 4791), it MUST also support creating calendar collections via extended MKCOL | — | deferred |
| R-12 | If server supports CardDAV address book creation, it MUST also support creating address book collections via extended MKCOL | `rfc5689.mkcol-addressbook-creates` | lint ✓ |

**Missing tests:**
- `rfc5689.mkcol-calendar-equivalent-to-mkcalendar` — calendar created via extended MKCOL has same `DAV:resourcetype` (`DAV:collection` + `CALDAV:calendar`) as one created via MKCALENDAR

*(R-11 is conditional on CalDAV suite being active; deferred until rfc4791 suite exists.)*
