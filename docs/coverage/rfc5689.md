> **Context:** RFC 5689 extends plain MKCOL (RFC 4918 ¬ß9.3) to accept a request body that specifies the new collection's resource type and initial properties. This is the standard way to create typed collections ‚Äî CalDAV calendars and CardDAV address books ‚Äî without requiring protocol-specific methods (MKCALENDAR). RFC 4791 ¬ß5.3.1 and RFC 6352 ¬ß6.3.1 both reference extended MKCOL as the preferred creation mechanism.

## ¬ß4 ‚Äî Integration with CalDAV and CardDAV

RFC 5689 ¬ß4 states that when a server supports both extended MKCOL and a protocol-specific creation method (e.g. MKCALENDAR for CalDAV), it MUST support using extended MKCOL as an equivalent alternative.

| Req | Requirement | Current Test | Status |
|-----|-------------|--------------|--------|
| R-11 | If server supports MKCALENDAR (RFC 4791), it MUST also support creating calendar collections via extended MKCOL | ‚Äî | not covered |
| R-12 | If server supports CardDAV address book creation, it MUST also support creating address book collections via extended MKCOL | ‚Äî | not covered |

**Missing tests:**
- `rfc5689.mkcol-calendar-equivalent-to-mkcalendar` ‚Äî calendar created via extended MKCOL has same `DAV:resourcetype` (`DAV:collection` + `CALDAV:calendar`) as one created via MKCALENDAR
- `rfc5689.mkcol-addressbook-equivalent` ‚Äî address book created via extended MKCOL has same `DAV:resourcetype` as one created via the server's native mechanism

*(R-11 is conditional on CalDAV suite being active; can be skipped if server does not advertise `calendar-access` in DAV header.)*
